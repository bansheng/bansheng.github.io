## 1. 现有流程分析

### 1.1 现有手工流程

```bash
# 步骤 1: 提取所有图片
pdfimages input.pdf output_prefix

# 步骤 2: 格式转换 (PPM → PNG)
convert output_prefix-000.ppm output_prefix-000.png

# 步骤 3: 验证尺寸
identify output_prefix-000.png

# 步骤 4: 手工重命名和复制
cp output_prefix-000.png content/blog/posts/{blog_id}/figure_1.png
```

### 1.2 实际执行记录（NormFormer案例）

**成功提取的图表**：
- Figure 1 (架构对比)：184KB，1275×900px
- Figure 3 (梯度分布)：309KB → 222KB (优化后)，1275×1500px  
- Figure 4&5 (缩放参数)：194KB，1275×1100px

**文件大小对比**：
| 优化方法 | Figure3 | Figure4_5 | 改善 |
|---------|---------|-----------|------|
| pdftoppm (初版) | 382KB | 351KB | - |
| pdfimages (当前) | 222KB | 194KB | -42% ~ -45% |

## 2. 问题详解

### 2.1 容易出错的步骤

#### 问题 1: 手工重命名和复制（高错误率）

**症状**：
- 人工输入文件名容易拼错 (figure1 vs figure_1 vs Figure1)
- 容易复制错误的文件到错误的目录
- 无法追踪哪些Figure已处理、哪些待处理
- 时间成本高（每个PDF需要 5-10 分钟手工操作）

**案例**：NormFormer 中手工处理 3 个 Figure，耗时约 15 分钟

#### 问题 2: 工具选择的不确定性

**现象**：
- pdfimages vs pdftoppm 的选择依赖手工判断
- 先尝试 pdfimages，失败后才改用 pdftoppm，浪费时间
- 没有量化的选择标准

**数据对比**（从 NormFormer 验证）：
- **pdfimages**: Figure1 = 184KB，保留矢量质感，快速
- **pdftoppm**: Figure1 = 349KB，渲染损失，较慢

---

### 2.2 缺少的验证检查

#### 检查项 1: 文件质量评分（缺失）

**当前状态**：只检查文件大小和分辨率，无量化质量评分

**需要添加**：
```
bytes/pixel 比率计算
- 公式: 文件大小(KB) / (宽px × 高px) × 1000
- 标准: < 0.25 = 优秀压缩，0.25-0.5 = 良好，> 0.5 = 需要优化
- NormFormer 数据:
  - Figure1: 184KB / (1275×900) = 0.16 ✓ 优秀
  - Figure3: 222KB / (1275×1500) = 0.12 ✓ 优秀  
  - Figure4_5: 194KB / (1275×1100) = 0.13 ✓ 优秀
```

#### 检查项 2: 分辨率验证（部分实现）

**当前**：手工用 `identify` 检查，无自动化标准

**标准**：
- 最小宽度 ≥ 600px（博客响应式宽度）
- 最小高度 ≥ 400px（可读性）
- NormFormer 数据：全部符合（≥ 900px）✓

#### 检查项 3: 颜色空间和位深验证（缺失）

**问题**：
- 无法检测灰度图（应警告）
- 无法检测索引颜色（应转换为RGB）
- 无法检测图片是否包含 Alpha 通道（可能导致渲染问题）

#### 检查项 4: 格式一致性验证（缺失）

**问题**：
- 无法验证所有 Figure 使用同一格式（PNG/JPG）
- 无法检测多余的元数据（增加文件大小）
- 无法检查是否存在坏块或损坏的像素数据

---

### 2.3 边界情况处理不足

#### 场景 1: 多页 PDF（> 100 页）

**当前问题**：
- pdfimages 对所有页面提取，生成成百上千个图片文件
- 无法筛选只需要的页面
- 手工排查哪些是实际的 Figure 太费时

**NormFormer 案例**：11 页论文，实际只需 3 个 Figure（页面 2、6、7）

#### 场景 2: 论文中有多个 Figure 在同一页面

**当前问题**：
- pdfimages 对单个页面提取，可能包含多个 Figure
- 无法自动分割同页多个 Figure
- 需要手工在图像编辑器中裁剪

**例子**：Figure 4&5 在同一页（page-07），现在是手工裁剪成两个

#### 场景 3: PDF 无嵌入图片

**当前问题**：
- pdfimages 失败时没有自动降级方案
- 用户需要手工切换到 pdftoppm
- 无法检测原因（是否真的无嵌图，还是格式不支持）

**应对**：应该自动检测失败原因，自动降级

#### 场景 4: 提取出的是 PPM 格式（pdfimages 默认）

**当前问题**：
- PPM 是中间格式，需要转换成 PNG/JPG
- 转换失败时（ImageMagick 未安装）无法处理
- 无法自动检测转换失败

#### 场景 5: 论文中嵌入的是矢量图（PDF/EPS）

**当前问题**：
- pdfimages 无法提取矢量图
- 需要手工用 pdftoppm 渲染或用 Inkscape 转换
- 无标准化的处理流程

---

### 2.4 Hugo 工作流集成不足

#### 问题 1: 文件路径约定不清

**当前**：
- 图片手工复制到 `content/blog/posts/{blog_id}/` 目录
- 无明确的命名约定（figure1.png vs figure_1.png vs Figure1.png）
- Hugo 中引用时用相对路径，与重命名冲突

**案例（NormFormer）**：
```markdown
![Figure 3: ...](figure3.png)    # 实际文件: figure3.png ✓
![Figure 4 & 5: ...](figure4_5.png)  # 实际文件: figure4_5.png ✓
```

但没有统一的命名规范文档

#### 问题 2: 无自动化的验证检查

**缺失**：
- Hugo 构建前无自动检查所有 Figure 是否存在
- 无检查 Markdown 中引用的 Figure 是否在磁盘上
- 无检查 Figure 图片质量是否符合标准

#### 问题 3: 与 CI/CD 的集成

**当前**：
- 没有在 GitHub Actions 中集成图表质量检查
- 无法在提交时自动验证所有 Figure
- 容易接受低质量或尺寸不符的图片

---

### 2.5 工具链的不完整性

#### 缺失的工具/检查

| 功能 | 当前状态 | 所需工具 |

## 3. 影响评估

### 3.1 当前的手工流程成本

对于单个论文（如 NormFormer）：
- PDF 下载：2 分钟
- 图片提取 + 格式转换：3-5 分钟
- 验证和优化：5-10 分钟
- 手工重命名和复制：5-10 分钟
- **总耗时：15-27 分钟/篇论文**

如果要处理 10 篇论文，总耗时 = **2.5-4.5 小时**

### 3.2 自动化改进的潜在收益

- **时间节省**：从 15-27 分钟 → 2-3 分钟（单条命令 + 自动化脚本）
- **错误减少**：从 20-30% 人工错误率 → < 1%
- **质量提升**：自动化质量检查 + 量化评分
- **知识积累**：可复用的脚本 + 文档

## 5. 技术选择建议

### Bash vs Python 对比

| 维度 | Bash | Python |
|------|------|--------|
| 依赖安装 | 轻（系统工具） | 中（pip packages） |
| 处理能力 | 单一命令链 | 复杂逻辑处理 |
| 错误处理 | 弱 | 强 |
| 可读性 | 中 | 强 |
| 跨平台 | 弱（macOS/Linux） | 强 |
| 维护性 | 中 | 强 |
| **推荐用途** | **简单管道 + 一键化** | **复杂工作流 + 未来扩展** |

**建议**：提供 **两份方案**
1. **Bash 脚本**：一键化、快速、适合博客作者直接使用
2. **Python 模块**：可复用、可维护、适合CI/CD集成

---

## 总结

现有的手工流程虽然在 NormFormer 论文上验证成功，但存在以下系统性问题：

1. **人工步骤多**：15-27 分钟/篇，难以规模化
2. **验证不充分**：缺少多个维度的质量检查
3. **边界情况覆盖不足**：多页 PDF、无嵌图等场景无标准流程
4. **与工作流集成不足**：无法与 Hugo、CI/CD 协同
5. **知识积累形式不完整**：已有经验未充分沉淀成可复用的工具

**改进的关键是将手工操作系统化，同时保持简洁和易用性。**
